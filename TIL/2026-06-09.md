## 배운 것

| # | 내용 |
|---|------|
| 1 | **`@Cacheable` 동작 원리** — Spring AOP 프록시가 메서드를 감싸서 캐시 HIT 시 메서드 실행 없이 캐시 값을 반환, MISS 시 메서드 실행 후 결과를 캐시에 저장 |
| 2 | **`@Cacheable` key SpEL 표현식** — `key = "#currencyCode"` (파라미터), `key = "#user.id"` (객체 필드), `key = "#a + '_' + #b"` (조합), `key = "'ALL'"` (상수 문자열) |
| 3 | **self-call에서 `@Cacheable` 무효** — 같은 클래스 내부에서 `this.method()` 호출 시 AOP 프록시를 우회하여 캐시가 동작하지 않음. 반드시 외부 빈에서 호출해야 캐시 적용됨 |
| 4 | **캐시 어노테이션 3종** — `@Cacheable` (캐시 저장+반환), `@CacheEvict` (캐시 삭제), `@CachePut` (강제 갱신, 메서드 항상 실행 후 캐시 저장) |
| 5 | **Redis vs Caffeine vs ConcurrentMap 선택 기준** — 변경 빈도 거의 없는 소량 참조 데이터(ISO 통화코드 등)는 로컬 메모리 캐시(`ConcurrentMap`)로 충분. Redis는 인스턴스 간 공유가 필요하거나 TTL 기반 갱신이 필요한 경우에 적합 |
| 6 | **기존 인프라 재활용** — `ConcurrentMapCacheManager`가 이미 `CacheConfig`에 설정되어 있었기 때문에 Caffeine 신규 의존성 추가 없이 `setCacheNames()`에 `"currencies"` 항목만 추가하여 해결 |

## 이슈 / 트러블슈팅

| # | 이슈 | 해결 |
|---|------|------|
| 1 | `CacheConfig.CURRENCY_CACHE` 상수를 처음에 package-private(`static final`)으로 선언하여 다른 패키지(`cache/`)에서 접근 불가 | `public static final`로 변경하여 해결 |
| 2 | LinePay `QRConfirm` 호출 시 `[CMCER015]` 에러만 찍히고 정확한 원인 로그가 없음 | `sendLinePayRequest`에 `try-catch(Exception e)`로 감싸 `ResourceAccessException` 로깅 추가 |
| 3 | `onStatus`, `body == null` 핸들러가 모두 통과되어 에러를 잡지 못함 | 원인: `RestClientInterceptor.logResponse()`가 500 응답에서 `response.getBody()` 호출 시 `IOException` 발생 → `onStatus` 도달 전에 예외가 먼저 터짐 |
| 4 | LinePay API URL에 이중 슬래시(`//`) 생성 — `BASE_URL`(trailing `/`) + `getURL()`(leading `/`) 조합 문제 | HTTP 클라이언트가 자동 정규화하여 실제 요청엔 단일 `/` 사용되나, 설정값 `BASE_URL` trailing slash 제거 필요 |
| 5 | LinePay 서버가 HTTP 500 반환 — `RestClientInterceptor` 수정 후 확인된 실제 근본 원인 | 500 응답 body 내용 확인 필요 (`RestClientInterceptor` 에러 스트림 처리 수정 후 재확인 예정) |
