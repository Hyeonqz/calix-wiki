## 배운 것

| # | 내용 |
|---|------|
| 1 | `DELIVERY_TIMEOUT_MS`는 `send()` 호출 후 성공/실패 확정까지의 총 상한 시간으로, `RETRIES × RETRY_BACKOFF_MS + REQUEST_TIMEOUT_MS`를 모두 포함하는 상위 타임아웃. 제약조건: `DELIVERY_TIMEOUT_MS >= REQUEST_TIMEOUT_MS + RETRY_BACKOFF_MS` 위반 시 Producer 생성 실패 |
| 2 | `RETRIES_CONFIG` 기본값이 Kafka 2.1+부터 Integer.MAX(2147483647)로 설정되어 있어서, 명시적으로 제한하지 않으면 `DELIVERY_TIMEOUT_MS`(기본 120초)까지 무한 재시도함 |
| 3 | `ENABLE_IDEMPOTENCE_CONFIG`(기본 true, Kafka 3.0+)가 활성화되면 네트워크 재시도 시 중복 메시지를 방지하며, 이때 `MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION`이 5 이하여야 순서 보장 |
| 4 | `ACKS_CONFIG`의 `"all"`(Kafka 3.0+ 기본값)은 ISR(In-Sync Replicas) 전체 복제 확인 후 ACK를 반환하여 메시지 유실 방지 |
| 5 | 아웃박스 패턴 도입 전 1차 방어로 Producer 설정만으로도 재시도 3회 + 1초 간격 + 10초 타임아웃 + 30초 총 상한으로 충분한 기본 내결함성 확보 가능 |
| 6 | 프로그래밍 방식 트랜잭션(`PlatformTransactionManager.commit()`) 이후에는 `@TransactionalEventListener(AFTER_COMMIT)` 대신 `@EventListener` + `@Async` 사용해야 함 — 수동 커밋 후에는 활성 트랜잭션 컨텍스트가 없어 `@TransactionalEventListener`가 동작하지 않을 수 있음 |
| 7 | `java-test-fixtures` 플러그인의 `testFixturesImplementation`은 `main` 소스셋의 의존성 체인을 자동 상속하지 않음 — `core-lib` 엔티티를 사용하려면 `testFixturesImplementation project(':core-lib')` 명시 필요 |
| 8 | `@Async` 스레드에서는 Hibernate Session이 없으므로 Lazy 관계 접근 시 `no Session` 에러 발생. `@Transactional(readOnly = true)`로 새 Session을 열어 해결. `REQUIRES_NEW`는 기존 트랜잭션이 없는 async 스레드에서는 일반 `@Transactional`과 동일하게 동작하므로 불필요 |
| 9 | Kafka `@KafkaListener`에서 `@Payload` 어노테이션: 명시하면 메시지 역직렬화 실패 시 즉시 예외 발생(validation 강제), 생략하면 Spring이 암묵적으로 메시지 본문을 파라미터에 바인딩하되 validation은 적용되지 않음 |

## 이슈 / 트러블슈팅

| # | 이슈 | 해결 |
|---|------|------|
| 1 | `@Async` 스레드에서 Lazy 로딩 접근 시 `LazyInitializationException`(no Session) 발생 | `@Transactional(readOnly = true)` 추가하여 새 Hibernate Session 생성 — `REQUIRES_NEW` 불필요 (async 스레드에는 기존 트랜잭션 없음) |
