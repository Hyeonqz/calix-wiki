## 배운 것

| # | 내용 |
|---|------|
| 1 | **ADR 문서는 코드베이스와 대조 검증이 필수** — As-Is 구조가 실제 코드와 일치하는지 Grep/디렉토리 탐색으로 확인. `module-api/shared/`의 `HmacAuthFilter`, `ApiGlobalExceptionHandler`가 `gateway.app`을 import하는데 영향 범위에서 빠져 있었음 |
| 2 | **배포 모듈 내 비즈니스 로직 배치 기준** — "배포 모듈에 비즈니스 로직을 두면 안 된다"는 원칙의 정확한 의미: **공유되는** 로직만 라이브러리 모듈로 분리. 단독 사용 + 공유 불필요 + 내부 레이어 구조 유지 시 배포 모듈에 두는 것이 타당 |
| 3 | **DDD 레이어드에서 컨텍스트 간 의존 규칙** — 같은 모듈 내 Bounded Context(`app/`, `pos/`, `web/`) 간 직접 참조 금지. 횡단 관심사(`shared/`)만 참조 가능. 컨텍스트 간 통신은 Kafka 이벤트 또는 공유 라이브러리 모듈 경유 |
| 4 | **서블릿 필터에서 예외를 throw하면 `@ControllerAdvice`가 잡지 못한다** — 서블릿 필터는 `DispatcherServlet` 밖에서 동작. `response.getWriter().write()`로 직접 응답을 작성하고 `return`하는 방식이 올바름 |
| 5 | **FilterRegistrationBean URL 패턴과 shouldNotFilter()의 관계** — URL 패턴은 서블릿 컨테이너 레벨, `shouldNotFilter()`는 필터 내부 추가 필터링. URL 패턴을 좁게 잡으면 `shouldNotFilter()` 로직이 무의미해질 수 있음 |
| 6 | **필터에서 Path Variable 추출 방법** — 필터 단계에서는 `@PathVariable` 바인딩 미수행. URI 직접 파싱(`request.getRequestURI().split("/")`)이 가장 간단 |

## 이슈 / 트러블슈팅

| # | 이슈 | 해결 |
|---|------|------|
| 1 | HmacAuthFilter가 실제로 동작하지 않던 문제 — URL 패턴이 `/endpoints/app/linepay/*`이고 `shouldNotFilter()`가 linepay를 제외하여 어떤 요청에도 HMAC 검증 미실행 | URL 패턴을 `/endpoints/app/*`로 변경, 세부 경로 필터링은 `shouldNotFilter()`에 위임 |
| 2 | RequestBody에서 transactionNo 추출 시 NPE — `objectMapper.readTree(body).get("transactionNo")`가 null 반환 → `.asText()` 호출 시 NPE | body가 비어있거나 해당 필드가 없는 경우 null-safe 처리 필요 |
