## 배운 것

| # | 내용 |
|---|------|
| 1 | 기존 Wrapper가 이미 `router.push(\`\${pathname}?\${querystring}\`)` 로 URL에 검색 파라미터를 반영하고 있었으므로, 복원 로직만 추가하면 되는 구조였음 — 기존 코드 분석이 구현 범위를 줄이는 핵심 |
| 2 | `useRef` 패턴으로 마운트 시 1회만 URL 파라미터를 파싱하면, `useMemo` 의존성 문제나 `searchParams` 변경 시 재계산 문제를 회피할 수 있음 |
| 3 | `URLSearchParams`로 배열을 직렬화하면 `[1,2]` → `"1,2"` (toString 호출)로 변환됨. 복원 시 `String(v).split(',').filter(Boolean).map(Number)` 패턴으로 역변환 |
| 4 | `useForm({ defaultValues })` 는 첫 렌더 시에만 초기값을 읽으므로, prop으로 전달받은 `initialConditions`를 `formDefaults`로 계산해 전달하면 자연스럽게 폼이 복원됨 |
| 5 | SearchBar의 기존 auto-click 메커니즘(`searchButtonRef.current.click()`)이 폼 초기값 기반으로 동작하므로, 폼 초기값만 올바르게 설정하면 별도 검색 트리거 없이 복원+재조회가 자동 수행됨 |
| 6 | `@DependsOn`은 `@PostConstruct`에서 Storages 데이터에 접근할 때만 필요. 런타임 호출만 하는 클래스는 빈 주입만으로 충분 — `@Order(HIGHEST_PRECEDENCE)`가 초기화 순서를 사실상 보장하지만 `@PostConstruct` 순서는 의존성 그래프에 따라 결정됨 |
| 7 | `ServiceInitializer`는 `PaymentCompanyStorages` 싱글톤에 DB 데이터를 적재하는 역할이고, 이후 생성되는 서비스들은 같은 싱글톤을 주입받아 캐싱된 데이터를 사용하는 구조 |
