## 배운 것

| # | 내용 |
|---|------|
| 1 | **k6 vs JMH 차이**: k6는 HTTP API 외부 부하 테스트 도구, JMH(Java Microbenchmark Harness)는 JVM 내부 메서드 수준 성능 측정 도구로 목적이 다름. 서비스 벤치마킹은 k6가 적합 |
| 2 | **폐쇄망 + k6**: 개발 서버가 외부 접근 가능(all open)하면 인터넷망 PC에서 k6를 설치해 바로 부하 테스트 가능. 폐쇄망이면 k6 바이너리를 내부로 반입해야 함 |
| 3 | **k6 setup() 활용**: `setup()` 함수는 테스트 시작 전 1회만 실행되어 VU가 공유할 데이터(거래번호 풀 등)를 준비하는 데 사용. PL(폴링) 고부하 테스트 시 TK+QR로 거래를 미리 생성해두고 VU에 분배 |
| 4 | **k6 actionFlag**: PL 응답의 `actionFlag` 값으로 결제 진행 상태 추적 가능 (0=대기, 1=간편결제사 요청, 2=QR수신, 3=완료, -1=실패/만료) |
| 5 | **GPG 저장소 필요 이유**: Ubuntu apt는 공식 패키지만 기본 인식. 서드파티(k6 등)는 GPG 키로 패키지 신뢰를 검증한 뒤 별도 저장소를 apt에 등록해야 설치 가능. 악성 패키지 치환 공격 방지 목적 |
| 6 | **`@EventListener` vs `@TransactionalEventListener`**: `@EventListener`는 `publishEvent()` 호출 즉시 실행 → `@Transactional` 내에서 발행 시 DB 미커밋 상태로 Kafka 전송될 수 있음. `@TransactionalEventListener(phase = AFTER_COMMIT)`은 트랜잭션 커밋 완료 후 실행을 보장 |
| 7 | **`fallbackExecution = true`**: `@TransactionalEventListener`는 기본적으로 활성 트랜잭션 없으면 이벤트를 무시함. `fallbackExecution = true` 설정 시 트랜잭션 외부에서 발행해도 실행됨 → 기존 D-MPM처럼 트랜잭션 외부에서 발행하는 코드와 호환 유지 |
| 8 | **CPM 성공 판단 기준**: `CpmHolder.failRef() == null`이 기존 컨벤션 (`approval != null`보다 명확). 컨트롤러도 동일 기준 사용 |
| 9 | **단일 발행 포인트 설계**: `switch` 분기 후 결과 변수로 받아 `failRef == null` 하나로 SUCCESS / USERPAYING→SUCCESS 케이스를 모두 처리. 분산된 발행보다 유지보수성 높음 |
